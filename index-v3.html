🔧 Rusk & Core

Intermediate

🌐 Kadcast P2P

Advanced

Kadcast is a structured UDP-based peer-to-peer broadcast protocol. It uses a Kademlia-like binary routing tree to direct message flow deterministically, reducing bandwidth by ~25-50% vs gossip protocols while making latency predictable and lower.

Repository: dusk-network/kadcast · Protocol port: 9000/UDP

How It Works

Gossip Protocol (Random)
Kadcast (Structured Tree)
  • Each peer gets a 128-bit ID computed as Blake2b(ip_octets || port)
  • Peers organized by XOR distance: d(x,y) = x XOR y
  • Each peer maintains k-buckets holding up to K most-recently-seen peers (LRU eviction)
  • Broadcast uses a tree-shaped dissemination — sender selects β delegates per bucket at decreasing height
  • RaptorQ FEC encodes payloads with overhead factor f=0.15, tolerating ~12% packet loss

Key Parameters

ParameterTypical ValueRole
K20–100Bucket size; affects routing state and lookup depth
α3Parallel queries during network discovery
β3Delegates per bucket during broadcast
f0.15FEC overhead (tolerates ~12% UDP packet loss)

Node Runner Notes

  • UDP port 9000 must be open inbound/outbound in your firewall
  • Set kadcast.public_address explicitly for NAT environments, or use 'N/A' for auto-detection
  • Routing table builds over a few minutes after startup

🔐 ZK & PLONK

Advanced

PLONK is a universal, updatable zk-SNARK proof system — Dusk's pure Rust implementation over the BLS12-381 elliptic curve. It enables proving computation correctness without revealing inputs.

Repository: dusk-network/plonk (628 stars)

Why ZK Proofs Matter in Dusk

  • Transaction validity — prove inputs exist, haven't been spent, and balance without revealing amounts
  • Citadel identity — prove KYC verification without revealing identity
  • XSC securities — enforce eligibility rules without exposing participant data
  • DuskVM contracts — native SNARK verification support

🛡️ Phoenix & Moonlight

Intermediate

Dusk supports two native transaction models that coexist at the protocol level. Phoenix provides UTXO-based private transactions; Moonlight provides account-based transparent transactions. Users convert freely between them via the Transfer Contract.

Phoenix — Private Transactions

Phoenix uses a UTXO model where each value unit is a note stored as a cryptographic commitment in a Merkle tree. Spending requires a ZK proof showing inputs exist, haven't been spent, and balance correctly — without revealing amounts or addresses.

  • Notes — UTXO-like containers; transparent or obfuscated (encrypted)
  • Nullifiers — deterministic values published when a note is spent (prevents double-spending)
  • View Keys — read-only keys for third-party auditing without spending ability
  • Stealth Addresses — one-time addresses per transaction output
  • Phoenix 2.0 — adds sender-data encryption for regulatory compliance (EU MiCA)

Repository: dusk-network/phoenix

Moonlight — Transparent Transactions

Account-based model added mid-2024 for exchange integration and higher throughput. Public addresses with visible balances — works like Ethereum.

Comparison Table

⚡ Consensus & Staking

Intermediate

Succinct Attestation (SA)

Dusk's PoS committee-based consensus provides deterministic finality — once a block is accepted, it cannot be reversed. Each round produces exactly one block in three steps:

Deterministic Sortition selects committee members: non-interactive, stake-weighted, unpredictable in advance, and verifiable by any node.

Security: honest majority of 2/3+ total staked DUSK.

📝 Smart Contracts

Intermediate

Dusk smart contracts are written in Rust, compiled to WASM, and executed by the Piecrust VM. The Forge framework provides a #[contract] proc-macro that eliminates boilerplate.

Prerequisites

# Install Rust nightly with WASM target
rustup install nightly
rustup target add wasm32-unknown-unknown

# Install build tools
cargo install wasm-tools
cargo install cargo-expand  # optional, for macro debugging

Basic Contract with Forge

#![no_std]
#![cfg(target_family = "wasm")]

extern crate alloc;

use dusk_forge::contract;

#[contract]
mod my_contract {
    use dusk_core::abi;

    pub struct MyContract {
        value: u64,
    }

    impl MyContract {
        pub const fn new() -> Self {
            Self { value: 0 }
        }

        pub fn init(&mut self, initial_value: u64) {
            self.value = initial_value;
        }

        pub fn get_value(&self) -> u64 {
            self.value
        }

        pub fn set_value(&mut self, value: u64) {
            self.value = value;
            abi::emit("value_changed", value);
        }
    }
}

Piecrust VM — Key Concepts

  • WASM linear memory IS the contract state — no key-value store like EVM
  • Only modified pages (deltas) are written back after execution
  • Failed calls revert all deltas automatically
  • Use BTreeMap instead of HashMap (no randomness in no_std)

Host Functions

abi::self_id()Get the contract's own ID

abi::emit(topic, data)Emit a contract event

verify_plonkVerify a PLONK zero-knowledge proof

🔌 w3sper SDK

Intermediate

w3sper is Dusk's official JavaScript/TypeScript SDK for building dApps — managing accounts, querying state, submitting transactions, and generating ZK proofs.

Installation

# Deno (primary)
deno add jsr:@dusk/w3sper

# Node.js / npm
npx jsr add @dusk/w3sper

Connect & Query Balance

import { Network, ProfileGenerator, AccountSyncer } from "@dusk/w3sper";

const network = new Network("https://nodes.dusk.network");
await network.connect();

const profiles = new ProfileGenerator(seed);
const profile = await profiles.default;

// Phoenix (shielded) address
console.log(profile.address.toString());
// Moonlight (public) account
console.log(profile.account.toString());

// Query public balance
const syncer = new AccountSyncer(network);
const balance = await syncer.balance(profile.account);
console.log("Balance:", balance.value);

Transfer DUSK

import { Transfer, Gas } from "@dusk/w3sper";

const transfer = new Transfer(profile)
  .to("targetAccountAddress")
  .amount(1_000_000_000n)   // 1 DUSK = 1,000,000,000 Lux
  .gas(new Gas({ limit: 500_000n, price: 1n }));

const tx = await network.execute(transfer);
console.log("TX:", tx.id);

Contract Interaction

import { ContractCall, Gas } from "@dusk/w3sper";

// Read-only query (no gas)
const result = await network.query(CONTRACT_ID, "get_value", null);

// State-changing call
const call = new ContractCall(profile)
  .contract(CONTRACT_ID)
  .fn("set_value")
  .args(42n)
  .gas(new Gas({ limit: 1_000_000n, price: 1n }));

const tx = await network.execute(call);

Event Subscription (RUES)

import { EventStreamer } from "@dusk/w3sper";

const streamer = new EventStreamer(network);
for await (const event of streamer.contractEvents(CONTRACT_ID)) {
  console.log("Event:", event.topic, event.data);
}

💱 DEX & Sniping

Advanced

PieSwap is the primary DEX on DuskEVM — a Uniswap V2-style AMM (constant product formula) with 0.30% swap fees.

AMM Mechanics

Constant product formula: x * y = k. When swapping, the AMM solves (x + Δx)(y - Δy) = k to determine output. Fee (0.30%) is added to reserves, rewarding LPs.

Key Separation (Consensus Key vs. Owner Key)

A Dusk stake is defined by two keys: the consensus key (used to sign and vote on blocks) and the owner key (which controls unstaking and withdrawing DUSK). By default, both roles use the same key — meaning if your node is compromised, an attacker could unstake and steal your funds.

Separating these keys is a critical security practice: the consensus key stays on the node (hot), while the owner key is kept in a separate, secure wallet (cold). This way, even if someone gains access to your node, they cannot touch your staked DUSK.

How to Separate Keys

1🔒Create Owner Wallet
→
2📋Get Owner Address
→
3💰Stake with --owner
→
4✓Verify
# Step 1: Create a SEPARATE wallet on a secure machine (not the node)
# This will be your "owner wallet" — keep it offline / cold
rusk-wallet create
# Back up the mnemonic phrase securely!

# Step 2: Note the owner wallet address
# From the owner wallet, get your address:
rusk-wallet balance
# Copy the address shown (this is your owner address)

# Step 3: On your NODE, stake with the --owner flag
# This tells the network: "use my node's consensus key to sign blocks,
# but only THIS separate address can unstake or withdraw"
rusk-wallet stake --amt 1000 --owner <owner-wallet-address>

# Step 4: Verify the key separation
rusk-wallet stake-info
# You should see different addresses for consensus and owner keys
Important Notes:
  • If you already staked without the --owner flag, you must unstake first, then re-stake with the flag.
  • The owner address should be from a wallet on a different machine — ideally a cold or air-gapped wallet.
  • Only the owner key can call unstake and withdraw — if you lose access to it, your funds are locked.
  • The consensus key can only sign consensus messages (proposals, validations, votes) — it cannot move funds.
